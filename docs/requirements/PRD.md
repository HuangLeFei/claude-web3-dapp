## 2. 项目概述

### 2.1 项目背景

传统投票系统存在诸多局限性：
- **中心化信任风险**：计票过程由单一机构控制，存在暗箱操作可能
- **审计困难**：投票记录易被篡改，缺乏公开透明的追溯机制
- **跨境障碍**：无法支持全球范围内的去中心化组织（DAO）治理需求
- **效率低下**：传统选举周期长，人工统计易出错

区块链技术的出现为构建去中心化、不可篡改的投票系统提供了技术基础。本项目旨在利用以太坊及兼容链的智能合约技术，构建一个安全、透明、可验证的投票基础设施。

### 2.2 项目目标

**核心目标**：
1. 构建一套完全去中心化的投票协议，确保投票过程的公开透明与结果的可验证性
2. 实现零信任的投票机制，无需依赖中心化中介机构
3. 提供灵活的投票配置能力，支持多种投票场景和规则

**量化目标**：
- 单次投票交易 Gas 费用控制在 150,000 gas 以内（约 $3-5）
- 智能合约通过 OpenZeppelin 及 Trail of Bits 安全审计
- 支持 EVM 兼容链的全网部署，覆盖主流 Layer 1 和 Layer 2 网络
- 合约代码开源，核心模块测试覆盖率 ≥ 90%

### 2.3 目标用户群体

| 用户类型 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| **DAO 治理者** | 协议升级决策、资金分配提案投票 | 透明性、可验证性、抗审查性 |
| **社区组织者** | 社区重大决策、选举社区代表 | 参与门槛低、规则灵活、结果公开 |
| **企业管理者** | 股东投票、董事会决策 | 合规性、隐私保护、审计能力 |
| **开发者** | 集成投票功能到 DApp、构建治理模块 | 易于集成、API 友好、扩展性强 |
| **普通选民** | 参与去中心化组织治理 | 操作简单、Gas 成本低、隐私保护 |

## 3. 功能需求详解

### 3.1 选民管理模块

#### 3.1.1 功能描述
提供选民身份注册、验证、状态管理的完整生命周期管理能力。

#### 3.1.2 用户故事

| 角色 | 故事 |
|------|------|
| 系统管理员 | 作为系统管理员，我希望能够设置选民资格验证规则（如持有特定代币、白名单地址），以便控制投票参与权限 |
| 合格选民 | 作为持有资格的用户，我希望能够查询自己的投票资格状态，并在投票前完成必要的身份验证步骤 |
| 开发者 | 作为 DApp 开发者，我希望能够通过合约接口查询选民状态，以便在前端显示正确的 UI 状态 |
| 审计员 | 作为第三方审计员，我希望能够查询历史选民注册记录和投票行为，以验证投票过程的合法性 |

#### 3.1.3 验收标准

- **P0**：
  - 支持基于钱包地址的选民注册机制
  - 确保每个钱包地址在同一选举中仅能注册一次
  - 投票后自动标记选民状态为"已投票"，防止重复投票
  - 提供查询接口，可返回任意地址的投票状态（未投票/已投票/无资格）

- **P1**：
  - 支持白名单管理模式，允许管理员批量添加/移除合格选民
  - 支持基于代币持有量的资格验证（如至少持有 100 枚治理代币）
  - 提供选民身份哈希机制，支持链下身份与链上地址的绑定验证

- **P2**：
  - 支持基于零知识证明的匿名投票资格验证
  - 支持二次方投票机制，允许用户按持有权重进行加权投票

#### 3.1.4 优先级
**P0**（核心必须实现）

---

### 3.2 提案与候选人管理模块

#### 3.2.1 功能描述
支持创建投票议题、管理候选人/选项、配置投票规则和时间窗口。

#### 3.2.2 用户故事

| 角色 | 故事 |
|------|------|
| 发起人 | 作为提案发起人，我希望能够创建一个新的投票议题，设置候选人名单、投票开始/结束时间，以便启动一次选举 |
| 管理员 | 作为系统管理员，我希望能够审核并批准/驳回发起人的提案，以防止恶意或不当内容的发布 |
| 参与者 | 作为潜在选民，我希望能够在投票前查看所有候选人的详细信息（如提案描述、链接、社交媒体），以便做出知情的决策 |
| 监控者 | 作为系统监控者，我希望能够查询所有活跃和历史提案的状态，以便进行统计分析和数据可视化 |

#### 3.2.3 验收标准

- **P0**：
  - 支持创建提案，包含：提案标题、描述、候选人列表（至少 2 个选项）
  - 支持设置投票开始时间（可延迟生效）和结束时间（强制结束）
  - 每个提案生成唯一 ID，支持通过 ID 查询提案完整信息
  - 提议创建后进入"待审核"状态，审核通过后才能公开

- **P1**：
  - 支持候选人元数据存储（如姓名、简介、头像 URL、提案文档链接）
  - 支持批量候选人导入（如从 JSON 文件或 IPFS 哈希加载）
  - 支持提案修改功能（仅在投票开始前允许）
  - 支持投票类型配置（单选/多选/排序投票）

- **P2**：
  - 支持提案标签和分类系统，便于检索和筛选
  - 支持提案间关联功能（如后续提案依赖前置提案结果）
  - 支持提案版本历史记录和回滚能力

#### 3.2.4 优先级
**P0**（核心必须实现）

---

### 3.3 投票执行模块

#### 3.3.1 功能描述
提供用户执行投票的核心逻辑，支持多种投票模式和行为记录。

#### 3.3.2 用户故事

| 角色 | 故事 |
|------|------|
| 选民 | 作为合格选民，我希望能够在投票窗口期内提交我的选择，并收到交易成功的确认，以便完成投票行为 |
| 选民 | 作为需要投多个选项的选民，我希望能够一次性为多个候选人投票（如董事会选举选 3 人），以提高效率 |
| 选民 | 作为偏好排序投票场景的选民，我希望能够对候选人进行排序（如第一选择、第二选择），以便实现更精细的民意表达 |
| 开发者 | 作为前端开发者，我希望能够通过事件监听获取实时投票动态，以便在 UI 上展示实时投票进度 |

#### 3.3.3 验收标准

- **P0**：
  - 支持单选投票，每个选民为指定提案选择一个候选人
  - 投票前校验：选民资格、投票窗口状态、是否已投票
  - 投票成功后触发 `Voted` 事件，包含：投票者地址、提案 ID、选择的候选人
  - 投票数据不可篡改地记录在链上

- **P1**：
  - 支持多选投票（如投票规则允许选择最多 N 个候选人）
  - 支持排序投票（Rank Choice Voting），允许按偏好顺序投票
  - 支持"弃权"选项，允许选民表达拒绝所有候选人的立场
  - 支持投票撤销功能（仅在投票窗口内且未计票前允许）

- **P2**：
  - 支持加权投票（如基于代币持有量的投票权重）
  - 支持委托投票（允许选民将投票权委托给代理人）
  - 支持部分投票（在多选场景下允许投少于最大数量的选项）

#### 3.3.4 优先级
**P0**（核心必须实现）

---

### 3.4 计票与结果发布模块

#### 3.4.1 功能描述
自动统计投票结果，提供实时或延迟的结果查询接口。

#### 3.4.2 用户故事

| 角色 | 故事 |
|------|------|
| 参与者 | 作为选民，我希望能够在投票结束后查看最终统计结果，了解每个候选人获得的票数和占比 |
| 参与者 | 作为希望实时跟进投票进展的选民，我希望能够在投票期间查看实时统计结果（如果提案允许），以便了解民意倾向 |
| 开发者 | 作为数据分析者，我希望能够获取完整的投票记录历史，以便进行深度分析和可视化 |
| 第三方验证者 | 作为外部审计员，我希望能够独立验证投票结果的正确性，而不需要依赖中心化服务 |

#### 3.4.3 验收标准

- **P0**：
  - 投票窗口结束后自动计算各候选人得票数
  - 提供查询接口返回提案最终结果：候选人 ID、得票数、百分比
  - 触发 `VotingEnded` 事件，包含最终结果摘要
  - 结果数据永久存储在链上，可被任意节点验证

- **P1**：
  - 支持实时结果查询（提案创建时可配置是否允许实时查看）
  - 支持按时间切片查询投票进展（如每小时投票数量统计）
  - 支持结果导出功能（如生成 JSON 格式的完整数据包）
  - 支持 The Graph 协议集成，便于高效历史数据查询

- **P2**：
  - 支持自动结果公布（计票完成后自动发布到 IPFS 等）
  - 支持多轮投票的自动晋级机制（如淘汰最低票数候选人后进入第二轮）
  - 支持统计分析接口（如投票参与率、弃权率等）

#### 3.4.4 优先级
**P0**（核心必须实现）

---

### 3.5 权限与安全模块

#### 3.5.1 功能描述
实现基于角色的访问控制、安全防护机制和紧急暂停能力。

#### 3.5.2 用户故事

| 角色 | 故事 |
|------|------|
| 系统所有者 | 作为系统管理员，我希望能够管理其他管理员的权限，授予或撤销他们对关键功能的操作权限 |
| 紧急响应员 | 作为安全团队，我希望能够在发现重大安全漏洞时立即暂停合约，防止恶意交易继续执行 |
| 开发者 | 作为智能合约开发者，我希望能够使用已验证的安全库（如 OpenZeppelin），以减少安全漏洞风险 |
| 审计员 | 作为安全审计员，我希望能够验证合约的权限控制逻辑是否正确，确保没有未授权访问路径 |

#### 3.5.3 验收标准

- **P0**：
  - 集成 OpenZeppelin `Ownable` 或 `AccessControl` 实现基于角色的权限管理
  - 关键函数（如创建提案、修改配置、紧急暂停）仅允许授权地址调用
  - 实现防重入保护（`nonReentrant` modifier）
  - 实现 `Pausable` 机制，支持紧急暂停关键操作

- **P1**：
  - 支持多角色权限系统（如 Admin、Proposer、Auditor）
  - 实现时间锁机制，高敏感操作需经过延迟期才生效
  - 支持权限变更事件日志，便于审计追踪
  - 实现合约升级代理（UUPS 模式），支持紧急修复和功能升级

- **P2**：
  - 支持多重签名钱包集成，关键操作需多人授权
  - 支持基于 DAO 治理的权限变更（通过投票修改管理员列表）
  - 实现自动安全监控告警（如异常交易检测）

#### 3.5.4 优先级
**P0**（核心必须实现）

---

### 3.6 事件与可验证性模块

#### 3.6.1 功能描述
通过完整的事件日志体系，确保所有操作可追溯、可验证。

#### 3.6.2 用户故事

| 角色 | 故事 |
|------|------|
| 第三方验证者 | 作为独立验证者，我希望能够通过链上事件重建完整的投票过程，以验证结果的正确性 |
| 监控者 | 作为系统监控者，我希望能够订阅特定事件流（如新提案、新投票），以便实现实时监控和告警 |
| 开发者 | 作为前端开发者，我希望能够使用 Etherscan 或 The Graph 查询历史事件，以便在 DApp 中展示完整的活动记录 |
| 合规审计员 | 作为法律合规人员，我希望能够导出完整的交易历史记录，以满足监管审查要求 |

#### 3.6.3 验收标准

- **P0**：
  - 所有关键操作触发链上事件：
    - `ProposalCreated`：提案创建
    - `VoterRegistered`：选民注册
    - `Voted`：投票执行
    - `VotingEnded`：投票结束
  - 事件包含完整上下文信息（操作者地址、时间戳、相关数据）
  - 事件索引参数化，支持高效链下查询（如按提案 ID 过滤）

- **P1**：
  - 支持批量事件查询接口，减少查询次数
  - 提供 The Graph Subgraph 配置文件，便于第三方索引
  - 事件日志包含链上哈希引用，便于关联相关文档（如 IPFS 上的提案详情）

- **P2**：
  - 支持事件签名验证，防止伪造事件
  - 支持事件归档功能，将历史事件数据存储到低成本存储层

#### 3.6.4 优先级
**P0**（核心必须实现）

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 投票交易 Gas 费用 | ≤ 150,000 gas | 单次投票操作（含存储） |
| 投票交易执行时间 | ≤ 12 秒 | 主网区块确认时间（L2 可更快） |
| 提案创建 Gas 费用 | ≤ 300,000 gas | 含候选人数据存储 |
| 结果查询响应时间 | ≤ 1 秒 | 链上读取操作 |
| 合约部署 Gas 费用 | ≤ 2,000,000 gas | 初始化部署成本 |
| 并发支持能力 | 不限 | 无状态并发支持 |

### 4.2 安全要求

| 安全领域 | 具体要求 |
|---------|---------|
| **智能合约安全** | 通过至少两家知名审计机构（OpenZeppelin、Trail of Bits 等）审计 |
| **防攻击能力** | 防重入、防整数溢出、防未授权访问、防抢先交易 |
| **密钥管理** | 敏感操作不存储私钥，使用硬件钱包或多签方案 |
| **隐私保护** | 符合 GDPR 要求，支持数据遗忘权（链下映射可删除） |
| **应急响应** | 具备紧急暂停机制，应急响应时间 < 1 小时 |
| **漏洞赏金计划** | 发布漏洞赏金计划，赏金金额 ≥ $50,000 |

### 4.3 兼容性要求

| 类别 | 要求 |
|------|------|
| **区块链兼容** | 支持 EVM 兼容链：Ethereum、Polygon、Arbitrum、Optimism、BSC、Avalanche |
| **钱包兼容** | 支持 MetaMask、WalletConnect、Ledger、Trezor 等主流钱包 |
| **标准兼容** | 遵循 ERC-20（代币）、ERC-721/ERC-1155（NFT）标准（如涉及） |
| **ABI 兼容** | 合约 ABI 标准化，支持 ethers.js、web3.js 等主流库 |
| **跨链能力** | 初期不支持跨链投票（同一投票仅限单链），二期规划跨链桥接 |

### 4.4 Gas 优化目标

| 优化项 | 目标 | 策略 |
|--------|------|------|
| 存储优化 | 减少 50% SSTORE 操作 | 使用 Packed Struct、位运算 |
| 循环优化 | 避免 Gas 消耗高峰 | 批量处理、限制循环次数 |
| 事件日志 | 事件数据精简 | 仅记录必要信息，详情存储 IPFS |
| 代理模式 | 降低升级 Gas 成本 | 采用 UUPS 模式而非 Transparent |
| 批量操作 | 支持批量注册/投票 | 单次交易处理多用户 |

## 5. 技术约束

### 5.1 技术栈选择范围

**智能合约开发**：
- **语言**：Solidity ^0.8.20（或更高版本，不使用实验性特性）
- **框架**：Hardhat 或 Foundry（优先选择 Foundry 以获得更快的测试速度）
- **库依赖**：
  - OpenZeppelin Contracts ^5.0（安全库）
  - OpenZeppelin Upgrades ^5.0（UUPS 升级模式）
  - 限制第三方依赖数量，仅使用经过社区验证的主流库

**前端开发**（可选，未来规划）：
- **框架**：React 18+ 或 Next.js 14+
- **Web3 库**：ethers.js v6 或 viem
- **状态管理**：Zustand 或 Redux Toolkit
- **UI 组件**：shadcn/ui 或 Ant Design

**测试**：
- **单元测试**：Foundry/Hardhat 测试套件
- **集成测试**：Fork 主网测试
- **模糊测试**：Foundry Fuzzer
- **形式化验证**：可选，使用 Certora 或 SMTChecker

### 5.2 第三方库限制

**允许使用的库**：
- OpenZeppelin（合约库）
- ERC-XXX 标准实现（经 EIP 审核）
- 数学和加密库（如 ecPair、Merkle Proof）

**禁止使用的库**：
- 未经过审计的第三方库
- 含有已知 CVE 的库版本
- 非开源或授权不明的库

### 5.3 部署环境要求

**测试环境**（当前可用）：
- **测试网络**：GenesisChain Testnet
- **RPC URL**：https://v2test.genesischain.io
- **Chain ID**：44
- **测试钱包**：0x122eFF57fbFa843cD6a776848CE2e69e3197427b

**生产环境目标**：
- **主网络**：
  1. Ethereum Mainnet（优先级 P0）
  2. Polygon PoS（优先级 P1）
  3. Arbitrum One（优先级 P1）
  4. Optimism（优先级 P2）
- **部署工具**：Hardhat Deploy 或 Foundry Script
- **验证工具**：Etherscan 自动验证
- **监控工具**：Tenderly、Forta

**环境配置要求**：
- 所有合约必须通过本地测试网验证后才能部署到测试环境
- 测试环境通过全面测试后才能部署到主网
- 部署流程需包含多签审批机制

## 6. 用户体验要求

### 6.1 交互流程

#### 6.1.1 选民投票流程

```
1. 连接钱包
   → 检查钱包网络兼容性
   → 提示切换到正确网络（如需要）

2. 查看投票资格
   → 自动查询当前地址的投票资格
   → 显示状态：合格 / 未注册 / 无资格

3. 选择提案
   → 展示所有活跃提案列表
   → 显示提案详情：候选人、投票时间、规则

4. 执行投票
   → 选择候选人/选项
   → 确认投票内容
   → 提交交易（预览 Gas 费用）
   → 等待交易确认

5. 查看结果
   → 显示交易成功确认
   → 允许查看实时/最终结果
```

#### 6.1.2 提案创建流程

```
1. 管理员身份验证
   → 连接管理员钱包
   → 验证管理员权限

2. 填写提案信息
   → 输入标题、描述
   → 添加候选人列表
   → 设置投票时间窗口

3. 预览与确认
   → 展示提案完整信息
   → 确认无误后提交

4. 等待审核
   → 提案进入待审核状态
   → 审核通过后公开
```

### 6.2 错误处理

| 错误类型 | 用户提示 | 技术处理 |
|---------|---------|---------|
| **钱包未连接** | "请先连接钱包以继续操作" | 检测 `window.ethereum`，触发连接 |
| **网络错误** | "请切换到支持的网络（如 Ethereum Mainnet）" | 验证 `chainId`，提供切换提示 |
| **无投票资格** | "当前地址不具备投票资格，请联系管理员" | 查询返回状态，显示友好提示 |
| **已投票** | "您已经在此提案中投过票" | 检查投票状态，阻止重复提交 |
| **投票已结束** | "该投票已结束，无法继续投票" | 检查时间窗口，禁用投票按钮 |
| **Gas 不足** | "钱包余额不足以支付 Gas 费用，请充值" | 预估 Gas 失败，显示错误提示 |
| **交易失败** | "交易失败：{具体错误信息}" | 捕获交易回执，解析错误原因 |
| **网络拥堵** | "当前网络拥堵，交易可能需要较长时间确认" | 显示区块 Gas 价格，建议等待 |

### 6.3 用户反馈

**交易确认反馈**：
- 提交交易后显示："交易已提交，请等待确认..."
- 交易成功后显示："✓ 投票成功！您为 {候选人名称} 投了一票"
- 提供交易哈希链接到 Etherscan

**加载状态反馈**：
- 关键操作（如查询结果）显示加载动画
- 长时间操作（如大批量注册）显示进度百分比

**成功/失败反馈**：
- 成功操作使用绿色成功提示
- 失败操作使用红色错误提示，并提供重试按钮

## 7. 风险与依赖

### 7.1 技术风险

| 风险 | 影响 | 可能性 | 缓解措施 |
|------|------|--------|---------|
| **智能合约漏洞** | 资金被盗、投票结果篡改 | 中 | - 多轮安全审计<br>- 漏洞赏金计划<br>- 形式化验证 |
| **Gas 费用波动** | 用户体验下降、投票成本高 | 高 | - 优先部署到 L2（Polygon、Arbitrum）<br>- 批量操作优化<br>- Gas 提示和预估 |
| **合约升级失败** | 系统不可用、数据丢失 | 低 | - 使用成熟的 UUPS 模式<br>- 升级前在测试网演练<br>- 准备回滚方案 |
| **预言机失效** | 无法获取外部数据（如涉及） | 低 | - 本项目不依赖预言机（纯链上投票）<br>- 未来如需外部数据，使用 Chainlink |
| **前端安全漏洞** | 钓鱼攻击、私钥泄露 | 中 | - 安全代码审计<br>- 使用受信任的 CDN<br>- 提示用户警惕恶意网站 |

### 7.2 业务风险

| 风险 | 影响 | 可能性 | 缓解措施 |
|------|------|--------|---------|
| **监管不确定性** | 合规风险、项目下线 | 中 | - 咨询法律顾问<br>- 支持数据遗忘权<br>- 去中心化治理结构 |
| **用户采用率低** | 项目无法维持运营 | 中 | - 降低使用门槛<br>- 提供清晰文档<br>- 与 DAO 合作推广 |
| **恶意投票行为** | 投票结果失真 | 中 | - 严格的资格验证<br>- Sybil 攻击防护<br>- 链下身份绑定 |
| **社区治理冲突** | 分叉、项目分裂 | 低 | - 清晰的治理规则<br>- 透明的投票机制<br>- 争议解决流程 |

### 7.3 外部依赖

| 依赖项 | 版本要求 | 依赖描述 | 备选方案 |
|--------|---------|---------|---------|
| **Ethereum/Polygon 等公链** | 稳定运行 | 基础区块链基础设施 | 多链部署 |
| **OpenZeppelin** | ^5.0 | 安全合约库 | 自实现关键组件 |
| **MetaMask 等** | 最新版本 | 用户钱包 | WalletConnect、硬件钱包 |
| **Etherscan** | API 可用 | 交易查询和验证 | 自建索引器 |
| **RPC 节点** | 可用性 ≥ 99% | 链数据访问 | 多 RPC 提供商冗余 |
| **IPFS（可选）** | 可用 | 大数据存储 | Arweave、Filecoin |

## 8. 验收与测试策略

### 8.1 测试范围

#### 8.1.1 智能合约测试

**单元测试**：
- 每个公共函数测试覆盖率 ≥ 90%
- 测试正常流程、边界条件、异常情况
- 使用 Foundry/Hardhat 编写测试用例

**集成测试**：
- 完整投票流程测试（注册 → 创建提案 → 投票 → 计票）
- 多角色交互测试（管理员、选民、审计员）
- 权限控制测试（验证未授权调用被拒绝）

**安全测试**：
- 重入攻击测试
- 整数溢出测试
- 未授权访问测试
- 抢先交易测试

**模糊测试**：
- 使用 Foundry Fuzzer 进行模糊测试
- 覆盖极端输入和随机组合

#### 8.1.2 性能测试

**Gas 成本测试**：
- 测量各函数的 Gas 消耗
- 验证是否符合性能要求
- 优化高 Gas 消耗的函数

**压力测试**：
- 模拟大量并发投票
- 测试合约在高峰负载下的表现
- 验证 Gas 限制和区块容量

### 8.2 测试环境

| 环境 | 用途 | 网络信息 | 数据 |
|------|------|---------|------|
| **本地测试网** | 开发和单元测试 | Hardhat Network (fork 主网) | 模拟数据 |
| **测试网络** | 集成测试和用户测试 | GenesisChain Testnet | 测试数据 |
| **主网络（生产）** | 正式使用 | Ethereum Mainnet / Polygon | 真实数据 |

### 8.3 测试数据

**基础测试数据**：
- 至少 10 个测试账户地址
- 至少 3 个不同提案（不同投票模式）
- 每个提案至少 5 个候选人
- 模拟投票参与率：30%、60%、90%

**边界测试数据**：
- 最大候选人数量限制（如 100 个）
- 最大选民数量限制（如 10,000 个）
- 最短/最长投票窗口（如 1 小时 / 30 天）

**异常测试数据**：
- 无效的提案 ID、候选人 ID
- 重复投票尝试
- 超出时间窗口的操作
- 未授权操作尝试
